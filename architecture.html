System Architecture · For Technical Review

One shared ERP data layer.
Four systems on top.

All four manufacturing systems operate against a single SQL Server data layer connected to MIE Trak ERP. No ETL, no duplication, no integration overhead — the architecture that lets the shop floor and management work from the same source of truth in real time.

System Philosophy

Designed from the shop floor,
not from a software diagram.

Each system was specified to solve a problem identified through direct operational involvement, then built to integrate with the ERP the business already runs on. The guiding principles below shaped every technical decision.

Single Source of Truth

MIE Trak is the system of record. Every application reads from — and where appropriate writes to — the same SQL Server database. No application keeps its own private copy of production data that could drift out of sync.

Real Time over Batch

Production events update the ERP at the moment they occur, not at end-of-shift. Dashboards read live on every page load. The cost of a stale snapshot — a wrong scheduling decision — is higher than the cost of a live query.

On-Premise & Resilient

Everything runs on factory infrastructure over the LAN with no external internet dependency. A dropped broadband line does not stop production. Engineering documents never leave the premises.

Shared Infrastructure

The Operations Platform's data collection feeds the Intelligence Platform directly. New capability is built on infrastructure already paid for — each system multiplies the value of the others.

Read-Safe by Default

Reporting and dashboard layers are read-only against the ERP database. Write paths are deliberate, narrow and validated — production data cannot be altered through a reporting surface.

Maintainable by the Engineer

A consistent stack — Python, Flask, SQL Server — across all systems, version-controlled in Git, so one engineer who understands both the technology and the production environment can maintain all of it.

Shared ERP Data Infrastructure

How data flows through the programme

Operator data written by the Manufacturing Operations Platform is read by the Manufacturing Intelligence Platform. The Digital Job Pack Platform and Document Intelligence Platform read part and document records from the same ERP. One data layer; four platforms.

Data Flow

A production event, end to end

Following a single shop floor action through the stack shows how the layers cooperate — and why management sees the result within the same working day, not the next.

Step 1 · Shop Floor
Operator acts on an Android terminal
An operator clocks onto a work order at their work centre. The Android app captures the operator, job, work centre and timestamp.
Step 2 · Integration Layer
Flask REST API receives the event
The terminal calls the Flask REST API on the factory Windows Server. The API validates the action and translates it into the correct ERP transaction.
Step 3 · Data Layer
MIE Trak / SQL Server updated in real time
The labour record is written to the MIE Trak SQL Server database immediately. There is no end-of-shift batch upload; the ERP reflects the live state of production.
Step 4 · Intelligence
Dashboards read the same record live
The Intelligence Platform queries that same database on every page load. The active session, its elapsed time and the operator's efficiency appear in the management view — no duplication, no second data pipeline.
Step 5 · Decision
Management acts within the shift
A supervisor sees an overdue work order or a loaded work centre as it develops and reschedules — a decision made during the shift, not retrospectively from yesterday's manual entries.

AI Integration

Where AI fits — and where it does not

Every system in this programme is delivered and in daily production use. AI fits in how they were built, not in how they run:

  • In the build — an AI model was used as a software development partner to accelerate implementation, under full engineering direction.
  • In production — the four platforms are deterministic and ERP-integrated, reading and writing real production data through validated paths. No AI sits in the production decision loop.

The result is software whose behaviour is predictable and auditable on the shop floor — built faster than it would have been alone, with the engineering judgement and the accountability remaining mine.

Engineering Accountability

How these systems were built

All systems in this programme were implemented using AI-assisted engineering, with an AI model used as a software development partner under full engineering direction and accountability.

  • Manufacturing requirements, operational workflows and business logic — designed by the engineer.
  • ERP integration — specified and validated by the engineer.
  • Testing and validation on the live floor — the engineer's responsibility.
  • AI accelerated software development; engineering judgement remained the engineer's throughout.

The engineer who operates the production environment remains accountable for maintaining and improving every system presented here.
